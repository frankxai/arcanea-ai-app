  On The Naming Question

  You're right to think about this. "Starlight Intelligence" as a phrase may have trademark holders in enterprise AI. But here's why
  you're safe:

  1. Your package is @frankx/starlight-intelligence-system — scoped to your npm namespace
  2. It's MIT open-source — you're not selling a competing enterprise product
  3. The mission is fundamentally different — personal creators, not enterprise sales
  4. "SIS" as abbreviation is generic enough

  If you ever want to be extra safe, you could evolve the name to something more distinctly Arcanean (like "Starlight Vaults" as the
  primary brand, with SIS as the technical system). But for now, you're building open-source tooling, not competing for enterprise
  contracts.

  On Why This Architecture

  The honest reflection:

  What's genuinely good:
  - File-first JSONL vaults — portable, readable, no database required
  - ~/.starlight/ for standalone, ~/.arcanea/starlight/ symlink for Arcanea users — serves both audiences
  - Horizon Dataset as public training data — genuinely novel and important
  - Session evals and pattern learning — this is what actually makes AI assistants better over time
  - Federation model (fork → contribute → upstream) — scales without gatekeeping

  What's still theoretical:
  - MCP server for vaults — not built yet, just planned
  - Family vaults — concept only, no implementation
  - Obsidian/NotebookLM integration — just export commands, no real plugins
  - Community contributions — zero external contributors yet

  What absorbed well from OSS:
  - Claude's memory.md pattern is the proven foundation (we extend, not replace)
  - mem0's graph concept adds value on top (planned, not built)
  - The eval/pattern framework is original — neither mem0 nor Letta does this

  What the community actually needs first:
  1. npx starlight init that creates ~/.starlight/ and seeds empty vaults
  2. npx starlight vault add that's as easy as a tweet to write
  3. A 30-second onboarding: install → init → start using
  4. Real platform adapter that reads vaults on session start in Claude Code

  Not the architecture docs. Not the vision pages. The 30-second experience.
