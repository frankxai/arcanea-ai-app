# Arcanea Website Evaluation

Date: 2026-03-28
Scope: `apps/web`, top-level public positioning, navigation, homepage, developer surface, and workflow hygiene
Standard: product clarity and ecosystem legibility at the level expected from a top AI lab or best-in-class creator platform

## Executive Judgment

Arcanea has real depth, but the public surface still behaves like several products speaking at once.

The strongest assets are:

- a distinctive philosophy and canon layer
- a broad ecosystem of packages, tooling, and creative surfaces
- enough technical substance to justify a serious platform story

The weakest assets are:

- route sprawl
- inconsistent category framing
- too many parallel identities competing on first contact

Current judgment:

- product ambition: very high
- ecosystem depth: very high
- public clarity: medium
- trust and focus: medium
- homepage discipline: medium-low
- developer surface coherence: medium

## Benchmark Standard

Arcanea should benchmark itself against three different qualities at once:

- `OpenAI` for product clarity and clear product-family separation
- `Anthropic` for trust posture, category structure, and disciplined navigation
- `Canva` for creator-first onboarding and fast understanding of what the product helps people do

Arcanea should not imitate any one of them directly. The target is:

- product-first comprehension within 5 seconds
- ecosystem legibility within 30 seconds
- mythic depth within 2 to 5 minutes for people who want it

## What Is Working

### 1. Arcanea has a real platform story

The product is not merely a chatbot and not merely a lore universe. It has enough product, content, and tooling surface to justify being framed as a platform.

### 2. The mythology is an advantage when correctly placed

The canon, Academy, Guardians, Gates, and library are not empty branding. They can function as retention, depth, and differentiation once the user already understands the product layer.

### 3. The repo and package footprint is strategically valuable

The monorepo plus ecosystem can support a strong "product plus developer ecosystem" story if governance becomes explicit.

## What Is Not Working

### 1. Arcanea still tries to introduce too many identities at once

A new visitor can still encounter:

- creator app
- creative multiverse
- mythology system
- developer ecosystem
- source-available tooling layer
- Academy progression framework

All of those may be true. They should not all compete equally on the first screen.

### 2. Route sprawl weakens trust

The current route surface is too broad for the main public IA. It makes the product feel less curated than it actually is.

### 3. The developer story is present but not fully governed

There is enough substance for packages, MCP, prompts, and protocols to matter, but they need a cleaner relationship to the creator-facing product.

### 4. Some copy still over-claims or blurs categories

Terms like `open source`, `creative multiverse`, `creative operating system`, `ecosystem`, and `platform` are each used in ways that are sometimes true but not always precise enough.

## Recommended Public Hierarchy

Arcanea should present itself in this order:

1. `Platform`
Arcanea is a creative intelligence platform for chat, creation, research, and world-building.

2. `System`
It combines product surface, philosophy library, and structured learning into one coherent creative system.

3. `Ecosystem`
For developers and advanced users, Arcanea also offers packages, MCP surfaces, prompts, and reusable tooling.

4. `Canon`
Its mythology and Academy structures are the depth layer that make the system singular.

## Homepage Standard

The homepage should answer these questions in order:

1. What is Arcanea?
2. What can I do here right now?
3. Why is it different from generic AI tools?
4. Where do I go if I want depth?
5. Where do I go if I want to build with it?

If a section does not help answer one of those questions, it should not be above the fold.

## Navigation Standard

Primary nav should stay constrained to:

- Chat
- Create
- Library
- Academy
- Community
- Developers
- Pricing
- About

Everything else should be secondary navigation, section-level navigation, search-discovered, or archived from the main product path.

## Developer Standard

The developer story should be framed as:

- build with Arcanea
- extend Arcanea
- integrate Arcanea

Not:

- decipher Arcanea

The developers page should cleanly separate:

- packages
- MCP and protocol surfaces
- repo map
- contribution paths
- public versus commercial layers

## Workflow Standard

No workflow should remain active if it points at stale apps, stale packages, or fictional deployment topology.

The correct rule is:

- quarantine first
- replace with explicit ownership and source-of-truth rules
- reactivate only when the workflow maps to current reality

## Next Directives

### Immediate

- finish normalization of `about-content.tsx`
- unify homepage language around platform, system, ecosystem, and canon
- keep developer language precise on what is public, source-available, or commercial

### Next

- reduce public route prominence and formalize a smaller IA
- add a proper developer ecosystem index
- define source-of-truth rules for every active workflow and satellite repo

### Later

- build a canonical homepage section that explicitly explains Arcanea's three layers
- create a route governance checklist for every new public page
- establish quarterly public-surface reviews

## Final Direction

Arcanea should not become smaller in substance.
It should become clearer in presentation.

The right move is not to suppress platform, ecosystem, protocols, or mythology.
The right move is to stage them properly so the user understands them in the right order.
